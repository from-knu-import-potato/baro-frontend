## 📋 1. 서비스 개요

'BARO'는 효율적인 재고 흐름과 스마트 운영 환경을 의미합니다.
본 서비스는 OCR 및 AI 기술을 활용하여 식자재 재고 관리 업무를 자동화하고,운영 효율성을 향상시키기 위해 기획되었습니다.
소규모 카페 및 식당 운영자가 거래명세서를 촬영하는 것만으로 입고 데이터를 자동 등록하고 재고를 관리할 수 있도록 설계된 AI 기반 재고관리 서비스를 제공합니다.

| 항목 | 내용 |
|------|------|
| 서비스명 | BARO (바로) |
| 슬로건 | 찍으면 바로, 분석과 예측도 바로! |
| BARO 의미 | Best Inventory · AI-Assistant · Recommendation · Outlook-Future |
| 핵심 컨셉 | 복잡한 기기 연동이나 수기 타이핑 없이 AI가 알아서 관리해 주는 점주 맞춤형 웹 기반 재고 관리 시스템 (SaaS) |

<br/>

### 🎯 타겟층
- 30~50대 소규모 음식점·카페·분식점 사장님 (1~3인 운영 구조)
- 직접 재고 관리와 발주를 담당하며 디지털 도구에 중간 수준으로 익숙한 사용자
- 매주 2~3회 식자재 입고, 월 식자재 비용 약 100~200만 원 수준
- 복잡한 재고관리 프로그램은 사용하지 않지만, 재고 문제는 계속 겪는 소규모 자영업자

<br/>

### ✨ 핵심 기능
1. **OCR 기반 입고 데이터 자동화 기능**
   - 거래명세서, 영수증 등의 이미지를 OCR로 인식하여 식자재 정보를 자동으로 디지털화
   - 품목, 수량, 가격 등을 자동 입력하여 수기 입력의 번거로움 제거
   - 입고 관리 시간을 단축하고 입력 오류를 최소화

2. **AI 기반 보관 및 발주 가이드**
   - 가게의 재고 데이터 및 소비 패턴을 분석하여 수요를 예측 & 발주 가이드
   - 적정 발주량을 자동으로 추천하여 과잉 재고 및 품절 방지
   - 유통기한 데이터를 기반으로 식자재 소진 우선순위 및 폐기 위험 알림 제공

3. **식자재 가격 변동 분석 및 시세 정보 제공**
   - 외부 API를 활용하여 도매/소매 식자재 시장 가격 데이터를 수집
   - 주요 식자재의 가격 변동 추이를 시각화하여 제공
   - 가격 상승/하락 트렌드를 기반으로 구매 시점 의사결정 지원

4. **초기 세팅 및 마감하기를 통한 식자재 재고 관리 자동화**
   - 메뉴·식자재·레시피 등 초기 세팅을 통해 가게 기준 데이터 구성
   - 마감하기에서 판매 메뉴 입력 시 레시피 기반으로 식자재 이론 사용량 자동 계산
   - 직접 사용한 재고를 별도 입력하여 실제 사용량과 이론 사용량 간 오차 분석 제공
   - 산출된 재고 데이터를 재고 현황, 발주 가이드, 수요 예측에 연계 활용

## 🚀 2. 추진 배경 및 개발 필요성

- **식자재 입고 및 기록의 비효율성**: 비정형 종이 거래명세서를 매번 수기로 입력해야 하는 업무가 점주들에게 극심한 피로를 유발 (평균 40분/회 소요)
- **수요 예측 부재로 인한 비용 손실**: 직관과 경험에만 의존하여 유통기한 초과 폐기 비용(평균 원가의 15% 이상)이 고정적으로 발생하거나 피크타임 재료 품절로 매출 기회 손실 반복
- **시장 시세 정보의 비대칭성**: 도소매 농수산 시장 당일 가격 변동을 객관적으로 비교할 수 없어 납품업체의 임의 공급가를 그대로 수용하는 정보 격차에 노출
- **발주 판단의 비과학성**: 경험에만 의존하여 과잉/품절 반복, 데이터 기반 발주 결정 시스템 부재

## 🏗️ 4. 프로젝트 구조

- `app/` : 앱 전체 레이아웃과 라우팅, 전역 스타일 초기화
- `features/` : 기능 단위 모듈 
- `pages/` : 실제 페이지 컴포넌트 
- `shared/` : 재사용 가능한 공용 리소스, UI 컴포넌트, 디자인 토큰, 유틸 등
- `widgets/` : 반복되는 레이아웃 컴포넌트 (사이드바, 헤더, 탑탭 …)
- 아래 폴더구조는 예시입니다.

```
public/
src/
├─ app/
│  ├─ layout/                  # 페이지 레이아웃 및 공통 UI 구조 (Header, Sidebar, Footer 등)
│  ├─ routes/                  # React Router 라우팅 설정 (route 정의 집중)
│  └─ styles/                  # 전역 스타일 및 App 초기화
│     └─ AppInitializer.tsx    # 앱 시작 시 필요한 초기화 로직 (토큰 확인, 전역 설정 등)
│
├─ features/                   # 도메인별 기능 단위 모듈 (비즈니스 로직 집중)
│  │                           # ⚠️ 규칙: 페이지 컴포넌트는 pages/에, 로직·UI는 여기에
│  ├─ auth/                    # 로그인·회원가입
│  ├─ dashboard/               # 대시보드 요약 위젯, 알림
│  ├─ invoice/                 # 거래명세서 촬영·업로드·OCR 연동
│  ├─ inventory/               # 재고 목록·상세·편집
│  ├─ order/                   # 발주 생성·이력
│  ├─ analytics/               # 소비 패턴·폐기율 통계
│  └─ settings/                # 업체 정보·알림 설정
│     │
│     └─ (각 feature 내부 구조 통일)
│        ├─ components/        # 이 도메인 전용 UI 컴포넌트
│        ├─ hooks/             # 이 도메인 전용 커스텀 훅
│        ├─ api/               # 이 도메인 API 호출 함수
│        ├─ store/             # 이 도메인 전용 상태 (필요 시)
│        └─ types/             # 이 도메인 전용 타입
│
├─ pages/                      # 라우팅되는 실제 페이지 컴포넌트
│  │                           # ⚠️ 규칙: UI·로직 없이 features/ 조합만 담당
│  ├─ LoginPage.tsx
│  ├─ DashboardPage.tsx
│  ├─ InvoicePage.tsx          # 거래명세서 등록
│  ├─ InventoryPage.tsx        # 재고 목록
│  ├─ OrderGuidePage.tsx       # 수요 예측 및 발주 가이드
│  ├─ AnalyticsPage.tsx        # 통계·분석
│  └─ SettingsPage.tsx         # 설정
│
├─ widgets/                    # 여러 페이지에서 반복 사용되는 조합형 UI
│  │                           # ⚠️ 규칙: 단일 도메인 종속 X, 2개 이상 페이지에서 사용
│  ├─ StockStatusCard.tsx      # 재고 현황 카드
│  ├─ AlertBanner.tsx          # 유통기한 경고 배너
│  └─ OcrStatusBadge.tsx       # OCR 처리 상태 뱃지
│
├─ shadcn/                     # Shadcn 컴포넌트 및 훅 전용 폴더 (수정 금지)
│
├─ shared/                     # 도메인 무관한 순수 공용 리소스
│  ├─ api/                     # Axios 인스턴스, interceptor, 공통 에러 처리
│  ├─ assets/                  # 이미지, 아이콘 등 정적 리소스

│  ├─ constants/               # 공용 상수
│  ├─ design-token/            # 디자인 토큰
│  │  ├─ colors.css
│  │  ├─ index.css
│  │  ├─ spacing.css
│  │  └─ typography.css
│  ├─ hooks/                   # 공용 커스텀 훅 (useDebounce, useLocalStorage 등)
│  ├─ lib/                     # 외부 라이브러리 설정 (queryClient 등)
│  ├─ store/                   # 전역 상태 (사용자 정보, 인증 토큰 등)
│  ├─ types/                   # 공용 타입 (API 응답 공통 포맷 등)
│  └─ utils/                   # 공용 유틸 함수 (날짜, 포맷팅 등)
│
├─ App.tsx
├─ index.css
└─ main.tsx
```

## 🔗 7. API 연동 구조

- **Base URL**: `https://api.baro.com/v1` (임시)
- **인증 방식**: JWT Bearer Token — Authorization 헤더
- **토큰 갱신**: refresh token 방식 / 만료 시 /auth/refresh 호출
- **에러 응답 포맷**: 미정
- **공통 처리**: Axios interceptor에서 401 감지 시 자동 로그아웃

<br/><br/>

## 🧠 8. 상태 관리 전략

- **서버 상태**: React Query — 캐싱, 동기화, 로딩/에러 처리. 모든 서버 데이터는 무조건 React Query로 관리합니다.
- **클라이언트 전역 상태**: Zustand — 사용자 정보, 인증 토큰, 사이드바 열림 여부, 모달 시스템 등 순수 UI 상태 및 전역 설정에만 사용합니다.
- **로컬 상태**: 단일 컴포넌트 내 UI 상태는 `useState` 사용
- **폼 상태**: React Hook Form으로 분리 관리

## 📏 10. 개발 컨벤션

### 네이밍
- 컴포넌트: PascalCase (`InventoryTable.tsx`)
- 훅: camelCase + use 접두사 (`useInventory.ts`)
- 유틸함수: camelCase (`formatDate.ts`)
- 상수: UPPER_SNAKE_CASE (`MAX_UPLOAD_SIZE`)
- API 함수: camelCase + 동사 (`fetchInventory`, `createOrder`)

<br/>

### 컴포넌트 작성 패턴
```tsx
1. import 순서: React → 외부 라이브러리 → 내부 모듈 → 타입
2. 타입 정의 (Props interface)
3. 컴포넌트 함수 (named export 권장)
4. 스타일 Tailwind 클래스
```

<br/>

### Git 커밋 컨벤션
- **형식**: `[gitmoji] [텍스트 태그]: [제목] (한 칸 띄우고) [본문]`
- **예시**: `✨ feat: 상품 목록 조회 기능 구현\n\n- React Query를 사용하여 서버 데이터 페칭 로직 추가`
- **주요 깃모지**:

## 🚨 13. 에러 처리 전략

## ⚠️ 14. 알려진 제약 / 주의사항

- OCR 결과는 항상 수동 검수 단계를 거치도록 설계 — 자동 확정 로직 추가 금지
- 시세 API는 평일 오전 9시 이후만 데이터 갱신됨 — 주말/공휴일 예외 처리 필요
- 이미지 업로드 최대 10MB 제한
- 하나씩 추가 예정

- 클래스 컴포넌트 사용 금지
- 특정 라이브러리 사용 금지 사항 준수

3. **"UI 생성 [이름]"**: `shared/components` 폴더에 `shadcn/ui` 스타일을 계승하는 범용 UI 컴포넌트 생성.
